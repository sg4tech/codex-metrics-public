# Segmented conversation-text analysis may produce better efficiency signals than structural metrics alone

Single-hypothesis file.

### H-018 — Segmented conversation-text analysis may produce better efficiency signals than structural metrics alone

- Status: `active`
- Created: `2026-04-02`
- Statement:
  - Segmenting conversation transcripts by work phase and analyzing those text windows may produce more useful efficiency and workflow-action signals than looking only at structural events, totals, and final outcomes.
- Why it matters:
  - If true, `codex-metrics` should not stop at reconstructing history from `~/.codex`; it should also evaluate how efficiently the conversation itself progressed through task clarification, planning, implementation, debugging, verification, and closure.
  - A narrow first script could also label each message with the most likely task-lifecycle action, such as continue the current task, start a new task, close a task, or reopen a task, so the history layer becomes a recommendation feed for task-boundary decisions instead of just a retrospective summary.
- Expected upside:
  - distinguish clean execution from noisy or drifting execution even when both tasks eventually succeed
  - surface conversation-level failure modes such as unclear task framing, repeated user corrections, unnecessary backtracking, and verification weakness
  - give agents more diagnosis-friendly signals than token totals or retry counts alone
  - make future workflow recommendations more specific because they can point to where the conversation got inefficient
  - provide a simple history-to-action script that can suggest the right task command type before the operator mutates the ledger
  - make start/close/reopen decisions easier to review because the transcript can be mapped back to explicit lifecycle intent
- Expected resource cost:
  - medium for an MVP segmentation and windowed analysis pipeline
  - high if we later add cross-history retrieval, evaluation datasets, or model-judged scoring loops
- Main risks or where this may be wrong:
  - conversation text may be too noisy or too prompt-heavy for stable scoring
  - efficiency judgments may become subjective unless the rubric is narrow and reproducible
  - a text-analysis layer could add cost and complexity without changing decisions materially
  - a naive whole-chat analysis could be expensive, unstable, and too context-heavy to trust
  - per-message command suggestions may overfit to surface wording instead of actual task state
  - a classifier that is not conservative enough could recommend reopen or close actions too often and create false positives
- Alternatives considered:
  - stay with structural metrics only
  - use one whole-chat LLM judgment per thread without segmentation
  - use RAG first and rely on retrieval instead of phase-aware segmentation
  - rely only on human judgment for start/close/reopen decisions and skip the recommendation layer entirely
- Current confidence:
  - `medium`
- Evidence status:
  - supported by the current local history, which already preserves full transcript text in session rollout JSONL files
  - supported by the limitation of structural metrics alone, which can show that a task was expensive or long without explaining whether the conversation was focused or confused
  - not yet validated by a real segment-scoring pipeline or by before/after evidence that conversation-text scoring improves recommendations
  - reinforced by the separate task-lifecycle workflow doc, which treats explicit stage and boundary decisions as a product lever rather than a bookkeeping detail
- Evaluation plan:
  - build conversation segmentation for major work phases such as task framing, planning, execution, debugging, verification, and closure
  - compute cheap structural features per segment before adding any LLM judging
  - add narrow windowed LLM evaluation on segment-sized transcript slices instead of feeding whole threads
  - compare segment-derived efficiency signals against existing ledger and history metrics to see whether they explain outcomes better
  - prototype a minimal message-level classifier that emits a suggested lifecycle command type per message or short window, starting with `continue`, `start-new-task`, `close-task`, `reopen-task`, and `uncertain`
  - compare classifier suggestions against manual retrospective boundary reviews on a small sample of real threads before any automation
  - only evaluate retrieval or RAG after the segmented baseline exists
- Next re-evaluation trigger:
  - after the first segment-level analysis pipeline is implemented and tested on several real threads
  - or after we find that structural metrics already explain enough variance that text-level analysis adds little value
