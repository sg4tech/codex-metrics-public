# H-040 — Session and message classification can be built from deterministic structural signals, making an LLM classifier optional rather than primary

Single-hypothesis file.

**2026-04-19.** Grounded in the investigation on `~/ai-agents-metrics-data/warehouse-full.sqlite` (262 anthropic sessions, 160 goals, 3.85B tokens). Direct consequence of H-015's need for a practice classifier — this hypothesis locks the mechanism.

### H-040 — Session and message classification can be built from deterministic structural signals, making an LLM classifier optional rather than primary

- Status: `active`
- Created: `2026-04-19`
- Statement:
  - The classified layer required for H-015 (practice-effectiveness analysis) can be populated from deterministic, non-LLM signals for essentially all practice types we care about today.
  - Session-level kind (`main` vs `subagent`) is 100% detectable from filename patterns in Claude Code history. Message-level kind (`skill_template` vs `human_authored`) is 100% detectable from `Skill` tool_use events in the parent assistant payload, or by matching a small set of stable template regexes.
  - An LLM classifier is needed only for a narrow residual: user-typed retrospective requests and intent markers where no tool_use signal exists. That residual is Codex-only in current data and covers one practice type (`retrospective`), not four.

- Why it matters:
  - H-015 is the central product thesis. Its cost, risk, and time-to-first-finding all depend on how we build the classifier. Going LLM-first adds tuning cycles, calibration overhead, and privacy considerations. Going structural-first removes all three for ~75% of the target practice types.
  - A structural classifier is reproducible byte-for-byte (same input → same output), which makes the `classifier_version` invariant in `warehouse-layering.md` trivially enforceable. An LLM classifier can only be reproducible with frozen models + deterministic decoding, and even then is expensive.
  - Quantifies the currently-hidden `attempt_count` aliasing bug from `docs/private/extraction-explained.md` §3: 174 of 262 anthropic sessions (66%) are subagent spawns mis-counted as user retries. Landing H-040 fixes that metric.

- Expected upside:
  - `derived_session_kinds` fully populated with 100% confidence on current data
  - `derived_message_kinds` with regex confidence ~100% on measured templates (32/32 matched templates all in main sessions, 0 in agent-prefixed — verified 2026-04-19)
  - `derived_goals.main_attempt_count` replaces `attempt_count` as the honest retry-pressure metric
  - Practice detection for `code_review_pass`, `qa_pass_requested`, `pr_hygiene`, `discovery` derivable directly from `Skill`/`Agent` tool_use names — no LLM call required
  - H-015 MVP scope reduced: LLM work limited to retrospective-intent detection, deferred to after structural-layer findings land

- Expected resource cost:
  - low-medium — requires: `derived_session_kinds` + `derived_message_kinds` schema, a classifier module that reads `raw_session_events` for tool_use extraction, config file listing known skill/agent names with practice mappings, migration path for updated `derived_goals.main_attempt_count`

- Main risks / where this may be wrong:
  - **Filename convention may change.** Claude Code's `agent-*.jsonl` / `subagents/agent-acompact-*.jsonl` naming is an internal convention that could break in a future version. Mitigation: treat filename match as one signal, not the only one — also cross-check with `Agent` tool_use events in the parent, and degrade to `kind='unknown'` when signals disagree.
  - **Compact subagent format is under-tested.** The `subagents/agent-acompact-*` variant (18 files, 7% of subagents) was discovered during this investigation but not structurally verified beyond filename match. May need handling we haven't seen yet.
  - **Codex symmetry is not verified.** Codex does not spawn subagents in the observed data — `session_kind='main'` for all 72 openai threads. Not a problem in itself, but if Codex ever adopts subagents the classifier must handle the pattern. Schema must not bake in Claude-only assumptions.
  - **Retrospective detection gap.** On Claude, 71/72 retro triggers were in Codex messages, 1/48 in Claude. The structural approach does not detect retro intent in user-typed messages on either provider — only the downstream markers (Skill:code-review, Skill:mark_done, etc.) that happen after retro is decided. This means "did doing retros help?" still needs a separate LLM pass over user-typed content on Codex messages.
  - **Hypothesis collapses if Claude Code emits a cleaner explicit parent_id.** Then matching by timestamp becomes unnecessary. This would be a *simplification* of the proposal, not a refutation — flagged here to keep revisiting.

- Alternatives considered:
  - **LLM-first classifier over all normalized messages** (previous H-015 approach). Rejected — higher cost and lower reproducibility for practice types that structural signals already cover. Still valid as a fallback layer on top of this hypothesis.
  - **Regex-only over message content, ignore tool_use events.** Rejected — regex over templates works (demonstrated for `code_review_skill`, `mark_done_checklist`), but the `Skill` tool_use in the parent payload is a cleaner upstream signal and names the exact skill invoked (e.g. `commit-commands:commit-push-pr` vs a generic `commit` regex).
  - **Filename-only classification.** Rejected as sole signal — would miss skill invocations (which don't spawn files) entirely. Kept as the session-level signal, combined with tool_use extraction at message level.

- Current confidence:
  - `high` that session-level filename classification works at 100% on current data
  - `high` that Skill/Agent tool_use events can be extracted cleanly from `raw_session_events`
  - `high` that parent-child linkage via timestamp match + `tool_use_id` produces confident `parent_session_path`
  - `medium-high` that `Skill:<pack>:<name>` and `Agent:<subagent_type>` taxonomy maps cleanly to practice types; edge cases (generic `Agent:general-purpose`) need a decision
  - `medium` that the residual LLM layer for Codex retro-intent is a small, scoped, deferrable task rather than core H-015 work

- Evidence status:
  - warehouse-full audit on 2026-04-19 (see summary in chat + `docs/private/extraction-explained.md`):
    - 262 anthropic sessions break down as 88 uuid-named (`main`), 156 `agent-*` (subagent), 18 `subagents/agent-acompact-*` (subagent compact), 0 unknown — 100% classifiable by filename
    - thread `723c0a14-f9c7-...` : 13 `Agent` tool_use events in parent, 13 `agent-*.jsonl` child files, timestamp match within 3ms on every pair
    - `code_review_skill` and `mark_done_checklist` templates: 32/32 in main sessions, 0 in subagent sessions — confirms skill invocations are inline, not spawned files
    - **100% of current "retry" count on Claude data is subagent-aliased, not user-driven.** Verified 2026-04-19 in descriptive practice-pass: 88/88 Claude threads have exactly 1 main session; all `attempt_count > 1` is subagent spawning. Earlier "66%" phrasing was an initial lower bound (174 subagent files / 262 total sessions); the corrected number counts at the thread level and is 100%.
    - Codex threads: 72/72 have exactly 1 session (no subagents in Codex). Combined: 160/160 threads warehouse-wide have `main_attempt_count = 1`.

- Evaluation plan:
  1. **Schema.** Create `derived_session_kinds` and `derived_message_kinds` per the shape described in chat 2026-04-19 (session_path PK, kind, parent_session_path, parent_source, parent_tool_use_id, classifier_version). Covered by `warehouse-layering.md` Layer 3 rules.
  2. **Classifier config.** `config/classifier_v1.yaml` with: filename patterns for session kind, tool_use name→practice_type mapping, template regex set for message kind. Config hash goes into `classifier_version = 'v1-{sha8}'`.
  3. **Classifier module.** `src/ai_agents_metrics/history/classify.py`. Reads `normalized_sessions` + `raw_session_events`, writes `derived_session_kinds` + `derived_message_kinds` + `derived_practice_events`. No external network calls; fully deterministic.
  4. **Aggregate update.** Extend `derived_goals` with `main_attempt_count`, `skill_invocation_count`, `subagent_invocation_count`, `main_retry_count` (additive — existing `attempt_count` / `retry_count` stay for backward compat, see ARCH-018 and downstream migration).
  5. **Golden test.** Pick 3 threads (one with many subagents, one with many skills, one with neither), check classifier output matches hand-verified ground truth.
  6. **First H-015 finding on top of this.** Rebuild descriptive practice-pass with structural data instead of regex proxies. Check whether `code_review_pass` or `qa_pass_requested` correlate with anything measurable on current warehouse.

- Next re-evaluation trigger:
  - after step 5 lands — check whether hand-verified ground truth matches classifier output at ≥95%; if lower, investigate missed cases before building on top
  - after step 6 lands — if structural practice-pass produces a usable H-015 finding, deprioritize the LLM residual; if not, decide whether to escalate to LLM fallback or rework the taxonomy
  - if Claude Code releases a new file naming convention that breaks detection, reassess the filename-as-signal decision

- Related hypotheses:
  - **Enables H-015** (central thesis) — removes the LLM dependency from the primary practice types, reducing H-015 to a cheaper and more reproducible task
  - **Coordinates with H-018** (message-text analysis) — H-018 described text-analysis as a primary method; this hypothesis narrows the scope by showing most of the target practices need structural signals, not text analysis
  - **Superseded aspects of earlier H-015 framing** (LLM-first over messages) — kept as fallback for user-typed retro intent, not dropped
  - **Blocks — not blocked by — ARCH-018** (raw-layer cleanup) — classified layer can be built on current normalized tables; ARCH-018 is an orthogonal cleanup
