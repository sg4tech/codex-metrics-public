Single-hypothesis file.

### H-020 — Workspace-wide history parsing may be required before cross-project analysis becomes reliable

- Status: `validated`
- Created: `2026-04-02`
- Statement:
  - `codex-metrics` may need to parse and retain history for all projects present in `~/.codex`, not only the current repository, so that analysis can compare behavior across projects instead of treating each project as an isolated slice.
- Why it matters:
  - If true, the code should preserve workspace and project identity as first-class analysis dimensions, because cross-project comparison is necessary to gather more varied data for future analysis, correlation search, and stronger conclusions about what actually works.
- Expected upside:
  - let agents compare quality, retry pressure, and cost across many projects in the same local environment
  - increase the diversity of evidence available for future analysis and correlation search
  - expose whether a workflow improvement is project-specific or broadly reusable
  - make the derived history more useful for “what changed across my projects?” questions
  - reduce the risk of overfitting recommendations to one repository's shape
- Expected resource cost:
  - medium for the first pass, because parsing all local project history likely needs broader discovery, storage, and indexing than a single-project slice
  - medium-to-high if we later need project grouping, filtering, and cross-project reporting surfaces
- Main risks or where this may be wrong:
  - expanding to all local projects may collect a lot more history than the current product can analyze well
  - cross-project data may be too heterogeneous to compare cleanly without a stronger classification layer
  - the extra surface could add complexity before we know which cross-project questions actually matter
  - “all projects” may be broader than the real decision boundary if only a subset of projects are relevant for analysis
- Alternatives considered:
  - keep the current single-project focus and compare only within one repository at a time
  - manually sample selected projects instead of parsing workspace-wide history
  - add cross-project analysis only at the report layer without broadening the underlying history model
- Current confidence:
  - `medium`
- Evidence status:
  - supported by the current project-review work, which already shows that project boundaries matter for interpretation
  - supported by the fact that cross-project comparison is already becoming a first-class analysis need in the framing
  - not yet validated by a workspace-wide pipeline that can reliably ingest, normalize, and analyze multiple project histories together
- Evaluation plan:
  - first expand `ingest-codex-history`, `normalize-codex-history`, and `derive-codex-history` so the workspace slice can cover all relevant local project histories
  - add or verify project and workspace identifiers in the history model so slices can be grouped across repositories
  - test whether analysis becomes more useful when it can compare multiple projects instead of only the current one
  - check whether cross-project views surface reusable workflow patterns that single-project analysis misses
  - confirm that the extra history volume does not drown out the signal we need for recommendations
- Next re-evaluation trigger:
  - after the first all-projects history slice exists
  - or after a cross-project question cannot be answered from the current single-project history model
