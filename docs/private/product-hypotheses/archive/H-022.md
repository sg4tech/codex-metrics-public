### H-022 — A retrospective timeline layer may be required before we can evaluate whether retrospectives improve later product outcomes

- Status: `archived`
- Created: `2026-04-03`
- Statement:
  - `codex-metrics` may need a dedicated retrospective timeline analysis layer that reads history in time order, marks retrospective events on that timeline, and computes comparable metric windows before and after each retrospective, because the current ledger and day-level aggregates are too coarse to evaluate whether retrospectives improve later product outcomes.
  - in the current product framing, this timeline layer exists to help an analyzing agent manage the effectiveness and economics of AI-agent-assisted engineering work, with a short verdict as the human-readable output
- Why it matters:
  - If true, the product should stop asking “are retros useful in general?” from coarse aggregate counts and instead build an explicit intervention-style dataset that lets an analyzing agent study `before -> retro event -> after` sequences.
  - This would turn a vague process belief into a concrete analysis surface for later statistical or quasi-experimental work.
- Expected upside:
  - make retrospective effectiveness measurable on a timeline rather than only discussable in chat
  - let agents compare product metrics in windows before and after a retrospective
  - create a reusable dataset for later significance testing, effect-size analysis, and failure-mode slicing
  - reduce over-reading of same-day or same-period `retro/meta` correlations
  - keep the analysis grounded in observed history instead of manual narrative reconstruction
  - support short verdicts on workflow changes instead of forcing a vague metrics recap, while staying aligned with the core product goal
- Expected resource cost:
  - medium for the first pass, because the feature needs event extraction, windowing rules, storage, and a read-only analysis command
  - medium if we later add richer retro classification, failure-mode tagging, or multiple windowing strategies
- Main risks or where this may be wrong:
  - the history may not expose retrospective boundaries cleanly enough for stable extraction without some live annotations
  - before/after windows may still be confounded by task mix, project changes, or overlapping interventions
  - a timeline layer may improve analysis shape without yet proving causal effect
  - poorly chosen windows could create a false sense of precision
- Alternatives considered:
  - keep using day-level aggregates for `retro/meta` vs `product` comparisons
  - evaluate retros only through manual archive review and qualitative judgment
  - add more live bookkeeping first instead of building a historical timeline layer
  - model only codified follow-up artifacts and ignore the retrospective event itself
- Current confidence:
  - `medium`
- Evidence status:
  - supported by the current failure of coarse day-level analysis to say much about whether retrospectives improve later outcomes
  - supported by the existing history pipeline, which already gives us transcript order, timestamps, goals, attempts, and timeline events
  - supported by the validated observation that retros with codified follow-up are more useful than decorative retros, which implies that timing and downstream effects matter
  - not yet validated by an implemented timeline dataset that actually stores retrospective anchors and before/after metric windows
- Evaluation plan:
  - extract retrospective events from the existing ledger/history surface
  - define stable window rules such as `N closed product goals before` and `N closed product goals after`
  - compute metric snapshots for each retrospective anchor
  - store the result as a read-only analysis dataset
  - verify that an analyzing agent can answer “what changed after this retro?” without manual transcript spelunking
- Next re-evaluation trigger:
  - after a first `retro-events` or equivalent analysis command exists
  - or after we test the first few retrospective windows and see whether they produce stable, reviewable output
