**2026-04-18 — Elevated to central product thesis.** This is now the primary question the tool exists to answer. Prior versions of H-015 framed this narrowly as "does retro/meta work improve throughput?"; that question is retained as the opening specific case and the strongest demand signal from the founder ("мне интересно, можем ли мы доказать, что ретроспективы приносят пользу"). The mechanism is locked to auto-classification of normalized session messages — no manual tagging. See `docs/private/product-strategy.md` amendment 2026-04-18 (final lock) and archived H-022 for the superseded ledger-based mechanism.

**2026-04-19 — MVP scope reduced to structural classification; LLM deferred.** Investigation on `~/ai-agents-metrics-data/warehouse-full.sqlite` (see H-040) showed that all four primary practice types (`retrospective` marker events, `qa_pass_requested`, `code_review_pass`, `discovery`) are detectable from deterministic structural signals in the warehouse — filename patterns for session kind, `Skill` / `Agent` tool_use events for practice events, and a small regex set for message-kind edge cases. LLM classification is downgraded from "primary mechanism" to "residual layer for user-typed retrospective-intent on Codex messages only." H-040 locks the structural mechanism; this hypothesis now depends on H-040 rather than owning its classifier design.

**2026-04-19 — Descriptive practice-pass returned a strong null; evaluation plan needs a methodology upgrade.** Ran a 15-minute ad-hoc analytical pass on `warehouse-full.sqlite` before committing to building the classifier infrastructure. Two findings invalidate the original evaluation plan:

1. **Retry pressure has zero variance.** 160/160 threads (88 Claude + 72 Codex) have exactly 1 main session. All `attempt_count > 1` on Claude is 100% subagent-aliased (H-040 estimated 66%; actual is 100%). "Do retrospectives correlate with fewer retries?" — the original step-6 claim — cannot be tested because there is no retry signal to correlate against.
2. **Practice-presence is confounded by task size, not effectiveness.** Threads with `code_review` practice have **20× more total tokens** and **22× longer duration** than threads without. Discovery (`Explore` agent): 5× tokens, 17× duration. These are not efficiency signals — they mean "code-review fires on threads big enough to be worth reviewing." A naive split-comparison will overstate every practice's effect.

Consequence: building the classifier infrastructure (H-040 steps 1–5) before upgrading the evaluation methodology produces data that cannot answer the central question. Step 6 ("first claim") needs replacement with one of:
- **within-thread analysis** — when `Explore` fires in the first 20% of a thread, does the remaining 80% show better tokens/output-token ratio than in threads where `Explore` did not fire early? Task size held constant per thread.
- **matched-pair comparison** — for each practice-present thread, pair with 1–3 same-size practice-absent threads; compare normalized metrics.
- **project-level time trends** — did tokens/output-token drop after the user adopted skill X in project Y?

None of these are in scope today. Tracking this as the blocking question for H-015 step 6; the structural classifier (steps 2–4) is still worth building because it is the data-collection layer all three methodology upgrades need. The cost of this finding: ~15 min of SQL + Python over warehouse-full, zero schema changes, zero code written. Original confidence marker "low that the effect is strong enough to survive naive N=1 analysis without careful methodology" confirmed.

### H-015 — AI-collaboration practices statistically correlate with better outcomes, and auto-classified history can detect which ones

- Status: `active` (central thesis)
- Created: `2026-04-02` (original), rewritten `2026-04-18`
- Statement:
  - A set of AI-collaboration practices — retrospectives, QA-pass requests, code review passes, model selection, discovery-first framing, long-context vs short-context style — may correlate with measurable differences in outcome: cost per goal, retry pressure, goal-completion rate, time-to-completion.
  - If the correlations exist and are detectable with N=1-developer volume, the tool's primary user-facing value is producing those findings on the user's own history, with visible statistical caveats.
  - Auto-classification of history is the only acceptable data-collection mechanism (founder will not manually tag). As of 2026-04-19 (H-040), the classifier is **structural-first** — filename and tool_use signals over `normalized_*` rows — with an **LLM residual** layer reserved for user-typed retrospective intent on Codex messages. The original framing "LLM classifier over normalized session messages" is retained as fallback, not primary.

- Why it matters:
  - This is the product thesis. Everything else in the tool exists to support answering this question.
  - The founder has run the tool for ~1 month and the *only* learning was gained outside the tool (cheaper models = similar subjective quality). For the tool to justify its existence it must produce at least one practice-effectiveness finding the founder would personally share.
  - "Does retrospective X help?" is a universal developer question; if the tool answers it credibly for the founder, the pattern generalizes as a reputation asset (per product strategy Section 1).

- Expected upside:
  - one or more practices confirmed or rejected for this user with a visible confidence interval
  - reusable dataset of `practice_event → outcome_window` pairs that supports future comparative analyses
  - a shareable finding that functions as proof-of-work under the product strategy's reputation goal
  - downstream: practice-effectiveness leaderboard per project / per week / per model

- Expected resource cost:
  - medium-high — requires: trustworthy base metrics (Week 1 work), practice taxonomy, LLM classifier with calibrated agreement against human rating, windowing and confounder rules, a reporting surface
  - risk of wasted spend on classifier tuning if base metrics are still unreliable

- Main risks / where this may be wrong:
  - **N=1 statistical reach.** Single developer, single month of usable data per slice. Classical significance testing is weak; most "correlations" will be noise. The product may need to lean on effect-size estimation + explicit confidence framing rather than p-values.
  - **Confounders.** Task type, codebase familiarity, day-of-week, model choice, and practice presence are entangled. Naive before/after windows will overstate effects.
  - **Classifier quality.** LLM-as-judge for "was this a retrospective?" may miss subtle cases or over-fire on surface keywords. Requires human-rated calibration set before production use.
  - **Practice is not discrete.** "Code review" as a spectrum from one line of feedback to a structured pass; tagging as binary loses signal.
  - **Null results are valuable but unshippable.** If the tool correctly reports "we cannot distinguish practice X from noise at this volume," it is honest — but it does not produce the reputation asset the strategy requires.
  - **Dependence on fragile base metrics.** Current cost/token aggregation has a known critical bug (see strategy amendment 2026-04-18 and `docs/private/extraction-explained.md` when written). Garbage-in metrics invalidate any practice-effectiveness claim.

- Alternatives considered:
  - Ask the founder to self-rate each session on practice presence. **Rejected** — founder refuses manual tagging (H-036 confirmed).
  - Report only aggregate per-practice averages without attempting causal windows. Cheaper and more defensible, but fails to answer the central question.
  - Outsource practice classification to rule-based keyword matching. Cheaper and reproducible, but will miss semantic cases (a retrospective written in Russian, a code-review done without the word "review", etc.).

- Current confidence:
  - `medium` that practices do correlate with outcomes in this user's data at some detectable effect size
  - `low` that the effect is strong enough to survive naive N=1 analysis without careful methodology
  - `high` that auto-classification is the only viable data-collection mechanism for this founder

- Evidence status:
  - founder's qualitative judgment that some practices "feel" more useful than others, but no quantified backing
  - one month of dogfood data exists but has not been analyzed through this lens
  - base metrics (cost, tokens, retry pressure) have known reliability issues that must be resolved first

- Evaluation plan:
  1. **Fix base metrics** (Week 1 work): pin down cost/token aggregation bugs, document goal/attempt heuristic, audit the multi-model-session issue. Output: `docs/private/extraction-explained.md`.
  2. **Define minimum practice taxonomy.** Start with: `retrospective`, `qa_pass_requested`, `code_review_pass`, `discovery_before_implementation`. Each defined by observable message-level signals — **revised 2026-04-19**: each mapped to a concrete `Skill` or `Agent` tool_use name (or filename pattern) per the config in H-040 step 2. Taxonomy-to-signal mapping is the contract; LLM is not invoked at this step.
  3. **Build the structural classifier** per H-040 (filename → `derived_session_kinds`, tool_use → `derived_message_kinds` + `derived_practice_events`). Produce practice-event rows in the warehouse: `(thread_id, session_path, practice_type, confidence, anchor_message_index, classifier_version)`. **No LLM call in this step** — structural signals cover all four primary practice types on current data. This replaces the original "LLM classifier over normalized session messages" step.
  4. **Calibrate the structural classifier** against a hand-verified ground-truth set (3 threads per H-040 step 5, extended to ~20 threads). Required precision ≥95% before publishing any user-facing claim. An **LLM residual pass** is only added if the structural layer misses detectable retro-intent in user-typed Codex messages — this is a deferred sub-task, not a blocker for step 5.
  5. **Compute before/after windows** per practice-event — candidate windows: next N goals, next N days. Report descriptive statistics first; causal claims only after multiple windows replicate.
  6. **First claim**: answer "do retrospectives correlate with lower cost / higher completion rate in the following N goals?" with visible caveats. If this produces no signal either way, reframe or halt. If it produces a signal, the LLM residual for retro-intent detection becomes the next investment; if not, the classifier stays structural and cheap.

- Next re-evaluation trigger:
  - when `docs/private/extraction-explained.md` is written and base-metric bugs are triaged
  - when the first classifier pass exists and is calibrated on ≥20 sessions
  - when the first descriptive per-practice snapshot exists for the founder's own data
  - if at any step the founder's honest reaction is "still doesn't answer my real question", reshape the taxonomy before proceeding
