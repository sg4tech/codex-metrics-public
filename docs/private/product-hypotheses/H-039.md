## Hypothesis

When codex-metrics is used across multiple repositories, each repo builds its own warehouse from the shared `~/.claude`/`~/.codex` history. But AI agents starting a new session in any repo have zero context about prior work in other (or even the same) projects. A portable warehouse export would let agents load cross-project history without re-running the full pipeline or querying SQLite directly.

## Core idea

Add a CLI command `export` that dumps derived warehouse data into a self-contained, portable format (JSON). The output should be compact enough for AI agent context windows but complete enough to understand project history, cost, and retry patterns.

### What to export

Minimum viable export — derived tables that carry analytical value:

| Table | Include | Why |
|---|---|---|
| `derived_projects` | yes | Top-level aggregate: thread count, tokens, retry rate, date range |
| `derived_goals` | yes | Thread-level: sessions, attempts, retry count, model, dates |
| `derived_attempts` | yes | Session-level: source, provider, cli_version, model, time span |
| `derived_session_usage` | yes | Token breakdown per session: input, cached, output, model |
| `derived_retry_chains` | optional | Retry pressure per thread — useful for pattern analysis |
| `derived_timeline_events` | no | Too granular (mostly usage_event rows), low context value |
| `derived_message_facts` | no | Contains conversation text — privacy concern, large size |

### What to compute at export time

- `cost_usd` per session — join `model` with `model_pricing.json`, apply token counts
- `cost_usd` per goal (thread) — sum of session costs
- `cost_usd` per project — sum of goal costs

### CLI interface

```bash
# export current project warehouse
ai-agents-metrics export

# export with filters
ai-agents-metrics export --since 2026-04-01
ai-agents-metrics export --project /path/to/other/repo

# pipe to file
ai-agents-metrics export > ~/shared-context/project-a.json
```

## Expected upside

- AI agents in any repo can load a compact JSON with full history context
- Cost analysis becomes portable — no SQLite access needed
- Cross-project comparison becomes possible by merging exports
- Warehouse data (authoritative per H-038) becomes accessible outside the pipeline

## Main risks

| Risk | Mitigation |
|---|---|
| Export grows too large for AI context | Filter by date range, project; exclude granular tables |
| Model not available in derived tables | Prerequisite: ARCH-016 (propagate model to derived tables) |
| Cost calculation may be wrong for unknown models | Use same pricing logic as HTML report; show `null` for unknown models |
| Export format may need to change | Version field (`export_version`) allows forward evolution |

## Prerequisites

- **ARCH-016**: model must be propagated to `derived_session_usage` and `derived_goals` before export can include per-session cost

## Confidence

- `high` that the use case is real — every new Claude session starts without cross-project context
- `medium` on the exact export shape — needs iteration after first use
