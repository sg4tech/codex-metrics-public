# Retro: продуктовый ревью HTML-отчёта без запуска отчёта

**Date:** 2026-04-20
**Task:** продуктовый QA `render-html` (не ARCH-, не BUG-, обзорный заход)

## Situation

Пользователь попросил "перепроверь внимательно наш html отчет с продуктовой точки зрения". Я выдал ревью на ~10 пунктов, прочитав только исходник шаблона (`_report_template.py`) и `product-framing.md`. Реальный отчёт не запускал, data_json не смотрел, скриншот не снимал.

Пользователь пушбэкнул: *"так ты его запусти как QA в начале и потыкай в него, прежде чем какие-то выводы делать"*.

На втором заходе нашёл 4 P0-бага, которые первый проход пропустил полностью. И два из пунктов первого ревью оказались неверными.

## What Happened

### Первый проход (по шаблону)

Вывод из первого ревью:
1. P0: title "Codex Metrics" ← **верно, мелочь**
2. P0: ledger-first layout хоронит value prop ← частично верно но не главное
3. P1: нет success-rate chart ← верно, но тоже не главное
4. P1: `↑ worsening` контринтуитивно ← **НЕВЕРНО**, ↑ = cost растёт это стандартная конвенция дашбордов
5. P2: `ndjson` / `warehouse` бейджи жаргон ← мелочь
6. P2: "Practice Events" непонятное имя ← мелочь
7. Нет интерпретационного текста ← справедливо

Из этого списка **ноль пунктов** про то, что отчёт буквально врёт на данных.

### Второй проход (с запуском тулы)

За ~15 минут работы с CLI и данными нашёл:

| # | Баг | Как найден |
|---|---|---|
| P0-1 | Секция `WAREHOUSE` выводит ledger-данные | stderr CLI: `retry: ledger, tokens: ledger` |
| P0-2 | "No retries" green badge — ложный позитив | Тот же stderr + знание что warehouse-full имеет 160 тредов |
| P0-3 | Cost by Model Y-axis clipped 24× (шкала $5, бары $120) | Вытащил DATA_JSON, прогнал smartMax вручную |

| — | Visual overlap на outlier-лейблах | Скриншот через chromium |

Все P0 увидел только потому что **запустил инструмент и посмотрел числа**. P0-3 невозможно найти без данных: это баг в smartMax на unbalanced распределении. P0-4 невозможно найти без проверки pricing.json.

### Chromium-ветка

Побочный тред: хотел скриншот отчёта.

1. `chromium --headless --screenshot=/tmp/foo.png file:///tmp/foo.html` → stdout "28593 bytes written", файла нет.
2. `chromium-browser --screenshot=$HOME/foo.png file:///tmp/foo.html` → stdout "28593 bytes written", файл есть (28K), но **внутри страница "Your file couldn't be accessed — ERR_FILE_NOT_FOUND"**. Snap-chromium не может читать `/tmp/`.
3. `cp /tmp/foo.html ~/foo.html && chromium ... file://$HOME/foo.html` → работает.

## Root Cause

### Главная причина (5 Whys)

1. **Почему первый ревью пропустил P0-баги?**
   Потому что я не запускал отчёт — смотрел только шаблон и framing-doc.

2. **Почему я не запускал отчёт?**
   Задачу "продуктовый ревью" я разложил как "прочитать → сравнить с framing → выписать gap'ы". Режим "открыть → потыкать" не включился.

3. **Почему режим "потыкать" не включился автоматически?**
   Триггер *"проверь внимательно HTML отчёт"* не воспринялся как QA-задача. Слово "ревью" → код-ревью → чтение. Ни в CLAUDE.md проекта, ни в global, ни в memory нет правила *"если артефакт user-visible → сначала запусти и посмотри на него"*.

4. **Почему правило не кодифицировано, если очевидно нужно?**
   Потому что для code-ревью чтение действительно первично. Разница между "review PR" и "review user-visible artifact" не была явно разделена. А именно второе — это test, не analysis.

5. **Почему это опасно?**
   Потому что продуктовый ревью, сделанный только по исходникам, звучит авторитетно (`P0/P1/P2` таблица, ссылки на строки кода), но на деле пропускает именно те баги, ради которых ревью и делается: то, что видит пользователь.

### Theory of Constraints

Узкое место нахождения продуктовых багов в user-facing артефакте — **"я действительно посмотрел на артефакт"**. Никакой объём чтения шаблона не показал бы Y-axis clipping или missing opus — это свойства данных × шаблона, не свойство шаблона.

После того как тула запущена, баги выпадают за минуты. Значит bottleneck был структурный (не "надо лучше читать код"), а decision-level ("начал с неправильного инструмента").

### Chromium-причина

Snap-пакет chromium имеет AppArmor профиль, который по умолчанию блокирует `/tmp/` для чтения и пишет в свой sandbox при отсутствии прав на целевой путь. stdout `X bytes written to file /tmp/...` **не означает что файл там оказался** — строка логируется до проверки AppArmor.

## Retrospective

Комбинация факторов:
- "ревью" → "читать", а не "тыкать" — мышечная память от code review
- CLAUDE.md правило *"A successful build does not prove runtime correctness"* правильное по духу, но я его не распознал как применимое к HTML-отчёту
- Отсутствие явного шага "перед тем как писать вывод — открой результат"
- Для chromium snap: не проверил что stdout-report соответствует ls

Пользователь был прав с первого же пушбэка: "запусти как QA в начале". Фраза буквально описывает отсутствовавший шаг.

## Conclusions

**Правило:** если задача просит ревью/оценку/проверку любого user-visible артефакта (HTML-отчёт, графики, CLI-вывод, web-страница, скриншот, документ) — первый шаг запустить/отрендерить и посмотреть на результат. Чтение исходника — второй шаг, для объяснения *почему* что-то выглядит так.

Эвристика в help-словах: *"проверь"*, *"ревью"*, *"посмотри"*, *"оцени"* + user-visible output → всегда сначала запуск, потом анализ.

**Правило для chromium snap:** входной HTML и выходной скриншот держать в `$HOME/` (или другой directory вне snap-блокировки). `/tmp/` для chromium-snap не работает. После `--screenshot=` всегда `ls -la` целевого пути — stdout врёт.
