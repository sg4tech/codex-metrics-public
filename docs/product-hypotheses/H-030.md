# Event sourcing may eliminate git merge conflicts on the metrics file while preserving history and sync

Single-hypothesis file.

### H-030 — Event sourcing may eliminate git merge conflicts on the metrics file while preserving history and sync

- Status: `validated`
- Created: `2026-04-06`
- Statement:
  - `metrics/codex_metrics.json` is currently a mutable JSON file tracked in git. Each `codex-metrics` CLI invocation mutates it in place. This creates two structural conflicts:
    1. The OS-level `uchg` immutability flag (set by the application guard) blocks `git pull` / `git merge` because git cannot unlink and replace a protected file.
    2. Even without the immutability flag, concurrent agents or users updating the file on separate branches will produce JSON merge conflicts on every pull.
  - The root cause is that a frequently-mutating derived state (a metrics database) is stored as a single mutable file in a system designed for text-diff merges.
  - Switching to an **append-only event log** as the canonical storage eliminates both problems structurally:
    - Each CLI write appends a new event record to `metrics/events.ndjson` (newline-delimited JSON) rather than mutating a shared document.
    - `codex_metrics.json` is **removed from the repository entirely** — it is not replaced by a cached/derived file on disk.
    - Current state (goals, entries, summary) is derived in-memory at CLI runtime by replaying the event log. It is never persisted.
    - Append-only lines from two branches never conflict: git sees only new lines added at the end of the log.

- Decided design (2026-04-06, confirmed 2026-04-07):
  - **Canonical storage**: `metrics/events.ndjson` — append-only, git-tracked
  - **Event schema**: command-level events (1:1 with CLI commands). Each event payload contains the full GoalRecord + AttemptEntryRecord state after mutation. Event types: `goal_started`, `goal_continued`, `goal_finished`, `goal_updated`, `goals_merged`, `usage_synced`. Replay strategy: last-write-wins per `goal_id` — trivial and robust.
  - **`codex_metrics.json`**: removed from git (added to `.gitignore`), not replaced by any persistent artifact. Existing history accessible via `git log`.
  - **Derived state**: computed in-memory on every CLI invocation by replaying the event log; never persisted.
  - **Immutability guard**: removed entirely. `file_immutability.py` and all `uchg`/`chattr` logic deleted. No OS flag on `events.ndjson` — append-only by convention enforced via CLI, not OS.
  - **Validation**: two explicit layers — (1) at append time: validate that the event is structurally valid and the transition is permitted; (2) after replay: full business-rule validation on in-memory state (supersession graphs, referential integrity entries→goals, summary consistency).
  - **Migration of existing data**: deferred — event log starts fresh from cutover point. Past history remains accessible via `git log` on the old file.
  - **Test infrastructure**: `test_storage_immutability.py` deleted; `test_storage_roundtrip.py` rewritten for NDJSON; CLI integration tests switched from direct `codex_metrics.json` reads to `show`-command or replay; `conftest.py` uchg fixture removed.
  - **Linear issue**: CODEX-50

- Why it matters:
  - The current design makes multi-agent and multi-machine workflows unreliable. Any parallel work that ends in a `git pull` risks a blocked or conflicted merge.
  - The immutability guard (H-021) was intended to protect against accidental edits, but makes git workflow fragile as a side effect. Event sourcing removes the tracked mutable file entirely, making the guard unnecessary.
  - History and auditability are preserved naturally: the event log is the history.
  - Derived state is always consistent with the log — no possibility of drift between the file and what actually happened.

- Expected upside:
  - eliminate the class of git merge failures on `metrics/codex_metrics.json`
  - preserve full metrics history in a diff-friendly, append-only format
  - make multi-agent and multi-machine sync reliable without custom git tooling
  - remove the immutability guard entirely — no more `uchg` flag management

- Expected resource cost:
  - high — requires reworking the storage layer, CLI read/write path, and event schema
  - existing `codex_metrics.json` history is not migrated (accepted tradeoff)

- Main risks or where this may be wrong:
  - append-only log can still conflict if two agents append to the exact same line position in the same git push window (rare but theoretically possible)
  - replaying the full event log on every CLI invocation adds latency as the log grows; may need snapshot/checkpoint mechanism eventually
  - schema evolution: old events must remain readable under new event schema versions

- Alternatives considered:
  - **Remove uchg flag only** — eliminates the OS-level block but leaves JSON merge conflicts on every concurrent edit. Not a solution.
  - **Custom git merge driver** — reduces conflicts but requires per-machine setup and breaks under schema changes. Workaround, not a structural fix.
  - **External database (SQLite, etc.)** — solves sync differently (not via git), but loses git as the history and sync mechanism — incompatible with the sync goal.
  - **Separate metrics git branch** — preserves git history but adds tooling complexity and non-standard workflow.
  - **`codex_metrics.json` as gitignored cache** — considered and rejected: adds complexity without benefit; if derived state is not trusted as canonical, there is no reason to persist it at all.

- Current confidence:
  - `high` — direction validated through architectural analysis; design decisions confirmed by user
