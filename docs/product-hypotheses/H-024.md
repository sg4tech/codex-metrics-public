### H-024 — The public repository may need generated SEO-ready docs and landing pages from the start to make open-source distribution discoverable

- Status: `active`
- Created: `2026-04-04`
- Statement:
  - once `codex-metrics` has a public repository, it may need generated SEO-oriented public docs from the beginning, rather than only a minimal README, because a technically clean open-source release may still remain effectively invisible if the public surface does not explain the product in searchable, structured, and distribution-friendly language.
  - if true, the public-repo rollout should include an explicit docs-generation or page-generation layer for discoverability, not only code publication and contribution mechanics
- Why it matters:
  - If true, publishing the code safely is necessary but not sufficient.
  - The project would also need a deliberate discoverability layer so that users searching for AI engineering metrics, agent productivity tracking, cost analysis, or similar workflows can actually find and understand the project.
- Expected upside:
  - improve discoverability of the public repo through search-friendly docs and landing content
  - make the public value proposition understandable faster than a source-tree-first repo alone
  - reduce the chance that the project is technically public but practically undiscoverable
  - create a reusable docs surface for sharing, linking, and future public positioning
  - make contributor onboarding easier if public docs explain use cases, terminology, and install paths clearly
- Expected resource cost:
  - low-to-medium for a first pass if the generated surface is limited to a few canonical pages
  - medium ongoing cost if the public docs generation pipeline needs maintenance alongside code and product framing
- Main risks or where this may be wrong:
  - investing in SEO too early may be premature if the product and public audience are still unstable
  - generated SEO content can easily become shallow or spammy if it is not grounded in real public use cases
  - the project may need public-product clarity before it needs search optimization
  - public discoverability may matter less initially than clean packaging, contributor UX, and repository safety
- Alternatives considered:
  - publish the public repo with only a strong `README.md` and defer SEO-oriented pages until later
  - rely on GitHub repository metadata, release notes, and occasional sharing instead of a generated docs layer
  - create public docs manually rather than through a generation pipeline
- Current confidence:
  - `low-to-medium`
- Evidence status:
  - supported by the observation that a new public repository is not only a code-hosting artifact but also the first public product surface
  - supported by the practical risk that a repo can be open source and still hard to find or understand
  - not yet validated by real public traffic, contributor behavior, or a tested docs-generation workflow
- Evaluation plan:
  - define the first public discoverability goals for the repository launch
  - decide whether the initial public surface needs only a strong README or also generated docs pages
  - identify the core search intents worth serving, such as AI engineering metrics, agent performance tracking, retry pressure, and cost analysis
  - test whether a generated docs or landing-page layer materially improves clarity and discoverability compared with a README-only baseline
  - keep the public language grounded in actual product framing rather than generic SEO copy
- Next re-evaluation trigger:
  - after the first public repository draft exists
  - or after the first public docs/distribution plan is reviewed
