### H-023 — A public-first core with a private overlay may be the safest way to open-source `codex-metrics` without breaking internal iteration

- Status: `validated`
- Created: `2026-04-04`
- Statement:
  - `codex-metrics` may need to treat the public repository as the canonical home for the reusable product code, while keeping a separate private overlay repository for retrospectives, audits, local history artifacts, and other internal-only operating material, because turning the current private repository into a public one would expose too much internal context and make later bidirectional sync fragile.
  - if true, the right operating model is not “private source repo plus occasional public export”, but “public-first product core plus private internal layer with an explicit sync mechanism”
- Why it matters:
  - If true, repository topology becomes part of the product strategy rather than a packaging afterthought.
  - This would let the project open-source the code safely without leaking internal material, while still preserving the ability to accept outside contributions and carry them back into internal work.
- Expected upside:
  - reduce the risk of accidentally publishing retros, audits, logs, snapshots, local metrics state, or other sensitive internal artifacts
  - make the open-source boundary explicit instead of relying on ad hoc cleanup before each publication
  - let external contributors work against a clean public codebase rather than an internal operations repository
  - make inbound changes from open source easier to ingest back into internal work
  - reduce long-term merge pain compared with maintaining two nearly identical full repositories
  - create a more credible public distribution story if the project later wants outside adoption or contributions
- Expected resource cost:
  - medium for the initial split, because repository boundaries, file layout, CI, docs, and contribution flow all need to be clarified
  - medium ongoing cost if the private overlay needs regular adaptation to upstream public changes
- Main risks or where this may be wrong:
  - the private overlay may accumulate local assumptions that make syncing with the public core awkward
  - a subtree-style sync may still create operational friction if the public/private boundary is poorly chosen
  - some docs or policies may be partially reusable and partially sensitive, which creates classification work rather than a clean binary split
  - public-first may feel slower at first if internal work often starts from operational needs rather than reusable product needs
  - the project may over-engineer repository mechanics before confirming that outside contributions will actually happen
- Alternatives considered:
  - keep the current private repository as the source of truth and periodically export a sanitized public mirror
  - make the existing repository public after deleting sensitive files from the current tree but keeping the history
  - rewrite and sanitize the full private history, then publish that repository directly
  - keep the whole project private and publish only packaged artifacts, docs excerpts, or code snapshots when needed
- Current confidence:
  - `medium`
- Evidence status:
  - supported by the current repository shape, where reusable code lives alongside clearly internal material such as `docs/retros/`, `docs/audits/`, and local metrics/history artifacts
  - supported by the practical need for two-way movement: internal improvements should be publishable without repeated cleanup, and external changes should be ingestible without manual copy-paste
  - supported by the observation that `.gitignore` helps with future local state, but does not solve already tracked internal files or git history exposure
  - not yet validated by a real split, a stable public/private boundary, or an actual contribution flowing from public back into private
- Evaluation plan:
  - define the minimal public product core: code, tests, packaging, docs, and workflows that are safe and useful to publish
  - define the private overlay contents explicitly: retrospectives, audits, local snapshots, operational policies, and any private-only automation
  - choose and test one bidirectional sync mechanism, with `git subtree` as the current leading candidate over manual copy, duplicate repos, or `submodule`
  - run one real outbound flow from private work into the public core
  - run one real inbound flow from the public repository back into the private overlay
  - measure whether the resulting workflow is understandable, low-friction, and safe against accidental disclosure
- Next re-evaluation trigger:
  - after the first public repository is created and populated
  - or after the first attempted round-trip sync exposes where the boundary or mechanism is wrong
