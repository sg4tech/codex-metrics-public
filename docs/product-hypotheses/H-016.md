### H-016 — A structured event-logging layer may be required before we can confidently judge which product changes are actually working

- Status: `active`
- Created: `2026-04-02`
- Statement:
  - `codex-metrics` may need a structured event-logging layer, as a product observability layer, so we can later tell which product changes helped, which ones hurt, and which ones only looked useful because the current history is too coarse.
- Why it matters:
  - If true, the product should evolve from recording final goal outcomes only to also recording the events that explain why those outcomes happened, so we can distinguish real product effects from coincidental success.
- Expected upside:
  - make future product decisions based on observed workflow events instead of inference from end-state notes alone
  - let us measure guardrail quality with specific counts such as recoveries, blocked continuations, and heuristic false positives
  - reduce ambiguity when comparing “what worked” versus “what just happened to coincide with success”
  - improve analysis for changes that affect agent behavior rather than only final goal completion
- Expected resource cost:
  - medium for the first schema and reporting pass, because the event model needs to be explicit and durable
  - low to medium for the first narrow implementation if we start with only the highest-value event types
- Main risks or where this may be wrong:
  - logging too many events could create noise and make analysis harder instead of easier
  - a poorly designed schema could capture surface activity without the causal detail needed for product decisions
  - the added instrumentation could be more work than the decisions it enables if we over-scope it
- Alternatives considered:
  - keep relying on notes and final goal status alone
  - add only ad hoc debug output instead of durable structured logging
  - defer event logging until a later analytics phase
- Current confidence:
  - `medium`
- Evidence status:
  - current metrics are good enough for coarse outcome tracking but not enough to answer key guardrail questions with confidence
  - the handoff-guardrail work exposed the need to distinguish recoveries, blocks, and false positives instead of collapsing them into generic success notes
  - current history can support retrospective judgment, but it cannot yet support clean counter-level product analysis
- Evaluation plan:
  - define a small set of durable events for the highest-value workflow changes
  - track counts for recovery creation, mutation blocking, and heuristic false positives
  - verify that the added logging actually improves decision quality before expanding the schema
- Next re-evaluation trigger:
  - after a first event schema exists for the handoff guardrail or a similar workflow change
  - or after we find ourselves repeatedly unable to answer product questions from the current goal-only history
