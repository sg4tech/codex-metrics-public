### H-025 — `bootstrap` may need to become a project starter that can create a new workspace directory and optional non-codex starter setup

- Status: `active`
- Created: `2026-04-04`
- Statement:
  - `codex-metrics` may create more real workflow leverage if `bootstrap` evolves from “add codex-metrics files into the current repository” into a broader project-start command that can initialize a brand-new project in a user-specified directory, because the user value may begin before a repository already exists and may include several useful starter assets beyond codex-specific tracking.
  - if true, the right product surface is closer to “start a new engineering workspace here” than only “inject codex-metrics into the current repo”, with optional starter modules for common project setup that are useful even when they are not strictly part of codex-metrics itself
- Why it matters:
  - If true, the current bootstrap boundary is too narrow for the actual job the user wants done.
  - The product would then be competing on how quickly it gets a new project into a useful working state, not only on how well it adds metrics bookkeeping after the project already exists.
- Expected upside:
  - reduce startup friction when the user wants to begin a new project from zero in a chosen folder
  - let the product own the first-run workflow instead of assuming an existing repository is already prepared
  - make bootstrap more aligned with real user intent if the desired outcome is “start a good project here” rather than “add codex-metrics to what I already have”
  - create room for optional starter packs such as repo hygiene, basic docs, CI, linting, or other reusable setup pieces that improve the first day of a project
  - make codex-metrics easier to adopt if it arrives as part of a strong project-start baseline rather than as an extra later step
- Expected resource cost:
  - medium for a careful first version, because directory creation, repo initialization boundaries, optional modules, dry-run behavior, and rerun safety all become part of the public contract
  - medium-to-high ongoing cost if the product starts owning non-codex starter surfaces that need templates, compatibility maintenance, and clearer scope boundaries
- Main risks or where this may be wrong:
  - the broader starter workflow may dilute the product and make `codex-metrics` less focused
  - “useful starter things” can sprawl quickly into a template generator instead of a metrics tool
  - different stacks may need different defaults, which can create a maintenance burden and premature framework-specific branching
  - users may actually prefer composition with an external project template tool rather than one larger `codex-metrics` bootstrap
  - creating a new directory and project skeleton has a larger side-effect surface and stronger safety requirements than the current repo-local scaffold
- Alternatives considered:
  - keep `bootstrap` narrowly focused on codex-metrics and require the user to create the target repo or folder first
  - add only a `--path` style target-directory option while still limiting generated files to codex-metrics artifacts
  - split the concept into two commands, such as a narrow `bootstrap` and a broader `start-project`
  - integrate with external project generators instead of owning generic starter assets directly
- Current confidence:
  - `medium-high` that faster onboarding matters for public adoption
- Evidence status:
  - supported by the user request to start a project in a new folder rather than only bootstrap the current repository
  - supported by the user request for optional useful starter pieces that are not strictly about codex itself
  - supported by the broader product intuition that the earliest setup step may be where the most leverage sits
  - not yet validated by repeated user demand, a concrete starter surface definition, or evidence that a broader initializer outperforms a narrow bootstrap plus external templates
- Evaluation plan:
  - clarify the smallest high-value version of “start a new project” without turning the tool into a generic template engine
  - separate the core requirement of target-directory creation from the optional requirement of extra starter assets
  - identify which starter assets are broadly reusable and agent-agnostic enough to belong in the product
  - compare one-command project start against a narrower path-targeted bootstrap plus documented companion tools
  - validate reruns, partial scaffold states, `--dry-run`, and conflict handling before treating the broader initializer as safe
- Next re-evaluation trigger:
  - after the first concrete proposal for target-directory bootstrap semantics exists
  - or after another real request confirms or narrows the need for non-codex starter setup
- Related hypotheses:
  - related to [H-017](docs/product-hypotheses/H-017.md) because a broader initializer changes the practical decision flow from “track work” toward “help create the working environment first”
  - may eventually require updates to [docs/product-framing.md](docs/product-framing.md) if the product expands from a metrics-focused bootstrap into a more general project-start surface
