# `chatgpt-history` may benefit from a history-analysis and cleanup workflow that can normalize conversation titles and flag repairable history slices

Single-hypothesis file.

### H-001 — a ChatGPT/OpenAI history workflow may be useful if it can analyze exported conversation history, detect cleanup candidates, and safely normalize titles where supported

- Status: `active`
- Created: `2026-04-04`
- Statement:
  - a history-analysis workflow may be useful if it can inspect exported ChatGPT/OpenAI conversation history, group conversations by id, and surface the slices that look worth cleaning up
  - the most valuable first step may be analysis rather than mutation, because the available public OpenAI docs confirm stateful conversation primitives but do not clearly expose a public ChatGPT web-history rename API
  - if title normalization is possible, the workflow may benefit from deterministic title suggestions based on conversation content, id, or a canonical naming rule
  - if mutation is not supported, the workflow may still be useful as a read-only audit tool that produces cleanup candidates, rename suggestions, and explicit unsupported-operation warnings
  - the workflow may also help identify broken, duplicated, or partially imported history slices before anyone tries to repair them by hand
- Why it matters:
  - if true, the user gets a repeatable way to understand and organize ChatGPT history instead of relying on manual inspection
  - a clear analysis pass may reduce the temptation to guess at unsupported mutation paths
  - title normalization could make long chat histories easier to scan, search, and compare
  - a safe read-only mode would still be valuable even if direct history editing never becomes possible
- Expected upside:
  - make exported history easier to review
  - surface likely cleanup candidates without manual browsing
  - provide deterministic rename suggestions when a naming rule is available
  - keep unsupported history mutations out of the critical path
  - create a foundation for later import, reconciliation, or repair tooling if a real API or export format appears
- Expected resource cost:
  - low for a first read-only analysis pass
  - medium if the workflow needs multiple export formats, schema normalization, and title heuristics
  - potentially high if safe mutation support must be discovered through official APIs or carefully constrained adapters
- Main risks or where this may be wrong:
  - the public API may not support the cleanup action the user wants
  - exported history formats may be inconsistent enough that cleanup suggestions are only partially reliable
  - title heuristics may look helpful while actually being too generic
  - if the workflow mutates anything, it may create more risk than value unless the boundary is extremely explicit
  - the best user experience may turn out to be a report, not a write-back tool
- Alternatives considered:
  - keep the workflow purely analysis-only and never attempt mutation
  - focus only on export parsing and conversation reconstruction
  - build a separate manual rename proposal generator rather than any action-taking tool
  - treat the history as a local notes/archival problem instead of a cleanup problem
- Current confidence:
  - `low`
- Evidence status:
  - supported by the user’s request to analyze ChatGPT history and possibly clean it up by id
  - weakly supported by official OpenAI docs that show stateful conversation/history primitives, but not a confirmed public ChatGPT history rename flow
  - not yet validated by a concrete export schema, an official history-editing endpoint, or a working mutation path
- Evaluation plan:
  - define the smallest useful read-only history analyzer first
  - confirm which history formats are actually available to parse
  - test whether deterministic title suggestions are better than manual naming
  - verify whether any supported write-back path exists before designing mutation mode
  - keep the analysis and cleanup modes separate so unsupported writes remain impossible by default
- Next re-evaluation trigger:
  - after the first export format is identified and parsed successfully
  - or after an official supported write-back mechanism is confirmed or ruled out
- Notes:
  - `2026-04-04`: added as a separate external idea note to capture the ChatGPT/OpenAI history-analysis and cleanup concept outside the main `codex-metrics` product framing.
  - `2026-04-04`: the current working assumption is that read-only analysis is the safe default and that mutation support should only be added if the underlying API or export format truly supports it.
