# Retrospective: JSON-First Markdown-Optional Workflow

## Situation

`codex-metrics` had already shifted to an agent-first product model:

- JSON was the structured source of truth
- `show` had become the main agent-facing runtime surface
- markdown reporting still existed as a default generated artifact

That created a mismatch.

The product had already moved toward agents reading structured metrics and CLI output, but the default workflow still regenerated and tracked `docs/codex-metrics.md` as if it were a first-class required artifact.

## What Happened

The workflow was changed to `json-first`.

Default behavior now is:

- update `metrics/codex_metrics.json`
- do not regenerate markdown automatically

Markdown is still supported, but only as an explicit export:

- `codex-metrics render-report`
- `--write-report` on relevant mutating flows

The repository also removed the tracked default `docs/codex-metrics.md` artifact so the project would not keep a stale or misleading markdown layer by accident.

## Team Lead Review

Design review outcome:

- the change was made at the workflow boundary, not by weakening the source of truth
- JSON remained canonical
- markdown support was preserved as an optional export instead of being destructively removed
- the new `render-report` command keeps the export path explicit and testable

Architecture review outcome:

- the change stayed additive where it mattered
- report generation logic still has one owner
- command behavior is clearer because report writing is now opt-in instead of hidden side effect

## QA

QA checks performed:

- targeted tests for init/bootstrap/update/report behavior
- full `make verify`
- live `./tools/codex-metrics show`
- live `./tools/codex-metrics render-report --report-path <tmp>`

QA result:

- `156` tests passed
- the CLI still works correctly in the new default workflow
- optional markdown export works when requested

## Product Demo

Before:

- markdown regeneration was automatic
- tracked report churn happened even when the primary user was an agent reading JSON and CLI output
- the repo could keep a human-readable artifact mostly out of inertia

After:

- the default workflow stays on the agent-first surfaces
- markdown remains available when explicitly useful
- the product now spends default effort on the surfaces that match the actual user model

## Root Cause

The old default was a carryover from an earlier stage of the product.

At that earlier stage:

- a human-readable report was a useful default scaffold output

But after the framing evolved to agent-first analysis:

- the report had become partly duplicate overhead
- automatic regeneration no longer matched the main consumption path

So the problem was not that markdown export existed.
The problem was that markdown export still behaved like a required artifact.

## 5 Whys

1. Why did the workflow feel misaligned with the product model?
   - Because markdown was still regenerated by default even though JSON and CLI had become the main agent-facing surfaces.
2. Why was that a problem?
   - Because it created churn, token overhead, and a duplicate artifact that agents did not clearly need by default.
3. Why was markdown still default?
   - Because the workflow inherited assumptions from an earlier human-readable reporting stage.
4. Why not remove markdown completely?
   - Because the product still needed a safe way to test whether the export retained occasional value.
5. Why is `json-first, markdown-optional` the right intermediate step?
   - Because it preserves optional export capability while aligning default behavior with the current agent-first product model.

## Theory of Constraints

The constraint was no longer report rendering capability.

The constraint had moved to default workflow alignment:

- the product was still spending default effort on a secondary surface
- while the primary user model had already shifted elsewhere

So the correct move was not to improve markdown more.
It was to demote markdown from default artifact to explicit export.

## Conclusions

- The product now has a better default workflow for its actual primary user.
- Optional export is a better fit than mandatory regeneration for markdown at this stage.
- This should still be treated as an active experiment, not final unquestioned truth.

## Permanent Changes

- `metrics/codex_metrics.json` remains the required source of truth.
- `docs/codex-metrics.md` is now an optional export, not a default regenerated artifact.
- `render-report` is the explicit markdown export command.
- `--write-report` enables markdown output on mutating commands when needed.
- Bootstrap and init now default to `json-first` behavior.
