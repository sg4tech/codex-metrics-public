# Parallel Validation False Positive Retro

## Situation

Во время рефакторинга typed summary layer после успешного `update` я запустил `show` параллельно в одном batched execution и увидел расхождение в итоговых числах summary.

## What Happened

`update` напечатал уже новые значения summary, а параллельно запущенный `show` показал старое состояние и создал впечатление, что в скрипте есть баг консистентности.

После последовательного повторного запуска `show` выяснилось, что persisted state корректный, а различие было вызвано stale read во время конкурентного исполнения зависимых команд.

## Root Cause

Проблема была не в бизнес-логике metrics updater, а в неверной технике валидации:

- `update` и `show` были зависимыми командами
- я запустил их параллельно
- `show` мог прочитать файл до завершения записи `update`

Итогом стала ложная диагностика “баг в коде”, хотя ошибка была в процедуре проверки.

## Retrospective

Это хороший пример того, что даже корректный automation habit может стать источником шума, если не учитывать зависимости между командами.

Параллельные tool calls полезны для независимых чтений и проверок, но не подходят для последовательностей вида:

- `update -> show`
- `init -> update`
- любые команды, где вторая опирается на побочный эффект первой

## Conclusions

- Это не продуктовый баг и не дефект metrics model.
- Это validation false positive, вызванный stale read.
- Для metrics workflow нужно различать независимые проверки и зависимые CLI шаги.

## Permanent Changes

- Зависимые команды updater проверять только последовательно.
- Параллельно запускать только независимые чтения, lint/typecheck и несвязанные inspections.
- Считать расхождение между `update` output и параллельным `show` сначала проблемой validation procedure, а не сразу багом приложения.
